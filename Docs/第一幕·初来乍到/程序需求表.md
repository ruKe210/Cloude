
| 系统ID | 名称 | 一句话 |
|--------|------|--------|
| **PROG-1** | Meta 事件演出 | JSON 驱动剧情，自动换背景/立绘，事件中增减卡牌并写入持久化 |
| **PROG-2** | 卡牌战斗 | JSON 配置一场战斗（多己方/多敌方），从持久化读取玩家牌组开战 |

**现有代码可复用**：`NarrativeEngine` / `NarrativeSceneController` / `MetaGameSession`（Flag 与战斗跳转）、`BattleManager` / `CombatActionManager` / `Action` 队列、硬编码 `Strike/Defend/Bash`。

**当前缺口**：无「玩家牌库」持久化、无 JSON 卡牌/战斗/Skill 定义、Meta 无 `bg`/立绘/获弃牌节点、战斗仅单敌人且牌组写死在 `BuildStarterDeck`。

## PROG-1 · Meta 事件演出系统

### 目标

用 **JSON 演出脚本** 跑 Meta 事件（对话、选项、分支、进战斗）；演出过程中可 **获得或丢弃卡牌**；**背景与立绘随节点自动切换**；所有卡牌变动 **立即写入 PlayerRunState**（内存 + 可存档）。

### 功能范围

| 子能力 | 说明 | 现状 |
|--------|------|------|
| JSON 剧本驱动 | `line` / `choice` / `set` / `if` / `anchor` / `battle:` / `end` | ✅ 基础已有 |
| 背景自动替换 | 节点字段 `bg` → 加载 `Assets/Art/Act1/Background/{bg}.png` | ❌ |
| 立绘自动替换 | 节点 `speaker` + `portrait`（或按角色表默认）→ Meta 正面立绘 | ❌ |
| 获得卡牌 | 新节点类型 `grant_cards` 或 `set` 扩展：`addCards: ["card_strike", ...]` | ❌ |
| 丢弃卡牌 | `remove_cards` / 选项代价：`discardCards: ["card_todo"]` | ❌ |
| 持久化 | 变动写入 `PlayerRunState.OwnedCards`，战斗结束回 Meta 不丢 | ❌ |
| 战斗交接 | 保留 `MetaGameSession`：`battle:{battleId}` + `returnNode` | ✅ 基础已有 |

### JSON 扩展示例（草案）

```json
{
  "id": "n020",
  "type": "line",
  "speaker": "导师·老陈",
  "bg": "bg_desk",
  "portrait": "mentor_default",
  "text": "这是你的初始工牌技能包。",
  "next": "n021"
},
{
  "id": "n021",
  "type": "grant_cards",
  "cardIds": ["card_strike", "card_strike", "card_defend", "card_defend"],
  "next": "n022"
},
{
  "id": "n030",
  "type": "choice",
  "prompt": "多余的废牌要丢吗？",
  "options": [
    { "text": "丢掉「摸鱼」", "discardCards": ["card_slack"], "next": "n031" },
    { "text": "留着", "next": "n031" }
  ]
}
```

### 验收标准

1. 播放 `act1_arrival.json`（至少 S01~S02）时，切换节点会换 **背景 + 说话人立绘**。
2. 脚本中 `grant_cards` / `discard_cards` 后，`PlayerRunState` 中牌库数量正确。
3. 从 Meta 进战斗再回 Meta，**牌库与 Flag 一致**，不重置为硬编码初始牌组。

### 主要模块（建议路径）

| 模块 | 路径 |
|------|------|
| 运行态 | `Scripts/Run/PlayerRunState.cs` |
| 存档 IO | `Scripts/Run/RunSaveService.cs` |
| Meta 节点扩展 | `Scripts/Meta/Data/` + `NarrativeEngine` |
| 演出 UI | `Scripts/Meta/Presentation/NarrativeSceneController.cs` |

## PROG-2 · 卡牌战斗系统

### 目标

**一份 JSON 战斗配置** 自动生成一场战斗：配置 **多名己方单位**（含玩家）、**多名敌方单位**、敌人波次/Intent；玩家 **抽牌堆来自 PlayerRunState.OwnedCards**（含升级后的数值），不再 `BuildStarterDeck` 写死。

### 功能范围

| 子能力 | 说明 | 现状 |
|--------|------|------|
| 战斗 JSON | `battleId`、友方列表、敌方列表、背景、胜利/失败回 Meta 节点 | ❌ |
| 多敌方 | `BattleContext` 支持 `List<CombatEntity> Enemies`，目标选择、AOE | ⚠️ 仅单 `Enemy` |
| 多己方 | 队友 NPC（如老陈场外不参战；可选第二友方实体） | ❌ |
| 牌组来源 | 开战从 `PlayerRunState` 实例化 `CardInstance` 填入 DrawPile | ❌ |
| 卡牌 JSON | 卡牌定义表：`id` / `cost` / `effects[]` / `upgradeLevels[]` | ❌ |
| 敌人 JSON | 敌人定义表：`id` / `hp` / `intentPattern` / `skills[]` | ❌ |
| 战斗入口 | Meta `battle:Act1_Mentor` → 读 `Battles/Act1_Mentor.json` 加载 | ❌ |

### 战斗配置 JSON 示例（草案）

```json
{
  "battleId": "Act1_Mentor",
  "background": "bg_battle_office",
  "returnWin": "A1-B1-POST",
  "returnLose": "A1-B1-PRE",
  "allies": [
    { "ref": "player", "spawn": "slot_player" }
  ],
  "waves": [
    {
      "enemies": [
        { "enemyId": "MOB_NULLREF", "slot": "slot_enemy_1" },
        { "enemyId": "CHAR_ENEMY_DOC", "slot": "slot_enemy_2" }
      ]
    },
    {
      "enemies": [
        { "enemyId": "MOB_MERGE", "slot": "slot_enemy_1" }
      ]
    }
  ]
}
```

### 验收标准

1. 修改 JSON 中敌人种类/数量，**不改 C#** 即可改变战斗阵容。
2. 玩家在 Meta 中刚获得的牌，下一场战斗 **DrawPile 中能抽到**。
3. 支持同场 **≥2 个敌人** 存活、依次/同时 Intent；玩家可选目标（单体牌）。
4. 胜利/失败后正确回 Meta 对应节点，并写回 `PlayerRunState`（如有战后奖励牌）。

### 主要模块（建议路径）

| 模块 | 路径 |
|------|------|
| 战斗配置 IO | `Scripts/Battle/IO/BattleConfigLoader.cs` |
| 卡牌定义 IO | `Scripts/Cards/IO/CardDefinitionLoader.cs` |
| 战斗工厂 | `Scripts/Battle/BattleFactory.cs` |
| 多实体上下文 | 扩展 `BattleContext`、`BattleManager` |

## PROG-3 · 技能升级 + 开箱系统

### 目标

**Cloude 伙伴** 可升级多条 **Skill**；Skill 在 **打出卡牌时** 附加「意想不到」的额外效果（改伤害、改目标、抽牌、捣乱等）。**玩家卡牌** 可消耗资源 **升级**（改费用/数值）；通过 **开箱（Loot Box）** 随机获得新卡，结果写入 `PlayerRunState`。

### 功能范围

| 子能力 | 说明 | 现状 |
|--------|------|------|
| Cloude Skill 树 | JSON：`skillId` / `maxLevel` / `levelEffects[]` | ❌ |
| Skill 升级 UI | Meta 事件或专用节点 `upgrade_skill` | ❌ |
| 出牌挂钩 | 出牌结算前/后：`SkillResolver` 按已装 Skill 注入 Action | ❌ |
| 卡牌升级 | `OwnedCard.upgradeLevel`，读 `CardDefinition.upgradeLevels` | ❌ |
| 开箱 | JSON 池：`lootBoxId` + `weights[]`；节点 `open_loot_box` | ❌ |
| 持久化 | Skill 等级、卡牌等级、新卡入库均进 `PlayerRunState` | ❌ |

### Skill 配置示例（草案）

```json
{
  "skillId": "skill_auto_lint",
  "name": "自动修 Lint",
  "maxLevel": 3,
  "levels": [
    { "level": 1, "hook": "on_card_played", "effect": "bonus_damage", "value": 2 },
    { "level": 2, "hook": "on_card_played", "effect": "bonus_damage", "value": 4 },
    { "level": 3, "hook": "on_card_played", "effect": "random_target", "chance": 0.2 }
  ]
}
```

### 开箱配置示例（草案）

```json
{
  "lootBoxId": "box_onboarding",
  "entries": [
    { "cardId": "card_hotfix", "weight": 40 },
    { "cardId": "card_review", "weight": 35 },
    { "cardId": "card_slack", "weight": 25 }
  ]
}
```

### 验收标准

1. Meta 中升级某 Skill 后，下一场战斗出牌可见 **额外效果**（至少 1 种可观测）。
2. 升级某张拥有卡牌后，战斗中该牌 **费用或数值变化**。
3. 开箱后新卡进入 `OwnedCards`，下一场战斗可抽到。
4. Skill 等级、卡牌等级、开箱结果 **存档重载后仍在**。

### 与 PROG-1 / PROG-2 的关系

- **PROG-1** 通过脚本触发：`upgrade_skill`、`open_loot_box`、`upgrade_card` 节点。
- **PROG-2** 出牌时 **PROG-3** 的 `SkillResolver` 介入 Action 队列。
- 三者 **只读写同一份 `PlayerRunState`**，避免 Meta 与战斗各存一套牌库。

## 共享基础：PlayerRunState（优先于三大系统实现）

```csharp
// 概念结构，实现时放 Scripts/Run/
PlayerRunState
├── OwnedCards[]      // { cardId, upgradeLevel }
├── CloudeSkills[]    // { skillId, level }
├── Flags             // 与 NarrativeFlagSet 合并或引用
├── CurrentScriptId
└── CurrentNodeId     // 可选，用于存档续播
```

| 任务 | 说明 | 优先级 |
|------|------|--------|
| 定义 `PlayerRunState` | 三大系统唯一数据源 | P0 |
| `RunSaveService` | JSON 存盘 / 读盘（`Application.persistentDataPath`） | P0 |
| 替换 `MetaGameSession` 中仅 Flag 的局限 | 会话恢复时一并恢复牌库与 Skill | P0 |

## 建议实现顺序

```
PlayerRunState + RunSaveService
  → PROG-1（JSON 演出 + 背景立绘 + 获/弃牌）
  → PROG-2（卡牌/敌人/战斗 JSON + 多敌人 + 读牌库）
  → PROG-3（Skill 升级 + 出牌挂钩 + 卡牌升级 + 开箱）
```

第一幕内容（导师战、Cloude 协作、龙虾战）全部通过 **JSON 剧本 + JSON 战斗 + JSON 卡牌/Skill** 配置，**不再为单场战斗单独开程序需求**。
